Reading time : Less than 8 minutes Just as I mentioned in the last post, gidentify project has given me another work around. My plan was to pipe the output of the command line tool identify to the graphical user interface. I was using GTK+ and the available option for this was to use GtkTextView widget which can be used to display mutiline text. Unlike GtkEntry, which could only display a single line text at a time, GtkTextView seemed to be flexible. GtkLabel was another possibility, but it lacked user functionality like Editing. By the way my plan was to display the output of the command on the GUI, and I thought the system function will be just enough. When I manned system(), it has given me a sad news that the system() function would just return the exit status of the command passed to system() as the argument. I searched for an API for this purpose but I was unable to find one. So I figured out a code like this,

else    {        GtkTextBuffer output_buff=gtk_text_view_get_buffer(output_view);        char command = (char)malloc(512);        char output = (char)malloc(4096);        char temp = (char)malloc(128);        strcpy(command,“identify -verbose ”);        strcpy(output,“”);        strcpy(temp,“”);        strcat(command,file);        strcat(command,“ > temp.out”);        system(command);        FILE fp=fopen(“temp.out”,“r”);        while (!feof(fp))   {            strcat(output,temp);            fgets(temp,512,fp);        }        gtk_text_buffer_set_text( output_buff, output, (gint)strlen(output));        fclose(fp);        remove(“./temp.out”);}

As you can see, it used a file, a temporary file named temp.out in the current directory. The identify command will write the output to this file. Later on the file is read and the output is displayed via GtkTextBuffer widget. This code was clunky. Because, it uses a file. A file is a costly resource in computer terminology. And the delay caused in between the file write and read was pretty good that the user could feel a slower GUI for a faster CLI. And also if there is a file named temp.out in the current directory (or what ever I could imagine genpasswd n could generate) it will get overwritten on execution of gidentify. So I figured out another possible C for this problem,

else    {        GtkTextBuffer output_buff = gtk_text_view_get_buffer( output_view );        char command = (char)malloc(512);        char output = (char)malloc(4096);        char temp = (char)malloc(128);        strcpy(command,“identify -verbose ”);        strcpy(output,“”);        strcpy(temp,“”);        strcat(command,file);        FILE pipe_fp;        if (( pipe_fp = popen(command, “r”)) == NULL)   {            alert_function(“Pipe Error!”);        }        while (!feof(pipe_fp))   {            strcat(output,temp);            fgets(temp,512,pipe_fp);        }        gtk_text_buffer_set_text( output_buff, output, (gint)strlen(output));        fclose(pipe_fp);}

This code has some advantages over the last one. It doesn’t use a file at all but just a file pointer only. And I was able to run it from /usr/bin on my local machine. But it even possess a little delay on execution on popen. May be there is another possible C for this program that can pipe a line at a time and display via the GtkTextView widget. But factually the problem has got fixed for while, and the program worked fine… See you in the afterlife…Subin Sebastian
